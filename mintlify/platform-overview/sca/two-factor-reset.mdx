<Note>
Applies only to customers in an SCA-required region (EU). Every endpoint here
returns **`409`** for other customers.
</Note>

When a customer loses an enrolled factor (a new phone, a deleted authenticator),
they recover it with a **2FA reset**: an identity (liveness) check that, once
passed, clears the lost factor so they can re-enroll it. It's a poll-based flow:
start, poll until liveness passes, then complete.

All paths below are relative to `https://api.lightspark.com/grid/2025-10-13`.

<Steps>
<Step title="Start the reset">

```bash
POST /customers/{customerId}/sca/factors/reset

{ "factor": "TOTP" }
```

Returns **`201`** with a `resetId` and opaque liveness handles. Embed
`livenessAccessToken` in the verification SDK, or send the customer to
`verificationLink`. `expiresAt` bounds the reset window.

```json
{
  "resetId": "…",
  "livenessAccessToken": "…",
  "verificationLink": "https://…",
  "expiresAt": "2025-10-03T12:30:00Z"
}
```

</Step>
<Step title="Poll until liveness passes">
```bash
GET /customers/{customerId}/sca/factors/reset/{resetId}
```

```json
{ "status": "PENDING" }
```

Poll with a short backoff. Two things end the loop: `status` reaches
`LIVENESS_PASSED` (proceed to complete), or the reset's `expiresAt` passes.
`status` is passed through verbatim, so match `LIVENESS_PASSED` exactly and treat
every other value (including `PENDING`) as "not ready yet."

To keep the loop bounded, stop at `expiresAt`, and apply your own timeout if the
start response didn't include one. If the window closes before liveness passes
(for example the customer abandons or fails the check), the reset is over: start
a new one rather than polling indefinitely.
</Step>
<Step title="Complete the reset">

```bash
POST /customers/{customerId}/sca/factors/reset/{resetId}/complete
```

Returns `204` and clears the lost factor. Calling it before liveness has passed
returns `400`. The customer can then re-enroll via
[factor enrollment](/platform-overview/sca/factor-enrollment).
</Step>
</Steps>
